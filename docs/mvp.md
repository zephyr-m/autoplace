# AutoPlace MVP

## Цель

AutoPlace проектируется не как одноразовое тестовое задание, а как поддерживаемый MVP автомобильной платформы. Тестовое задание задает техническое ядро: каталог автомобилей, подписки на фильтры, фоновый матчинг и уведомления без дублей.

MVP должен показывать не только код, но и живую систему: каталог меняется, задания выполняются в очередях, события создают уведомления, админка показывает состояние, а инфраструктурные экраны помогают наблюдать, что происходит внутри.

## Продуктовый вектор

Базовая бизнес-идея: AutoPlace соединяет готовый спрос покупателей с продавцами автомобилей.

Коммерческий ориентир:

- покупатель сохраняет фильтр;
- в каталоге появляется подходящий автомобиль;
- система создает уведомление покупателю;
- в будущем это событие может стать лидом для продавца или дилера.

Платежный слой в ближайшей реализации не обязателен. Но модель данных, админка и доменные события должны проектироваться так, чтобы лиды, тарифы и биллинг можно было добавить без слома архитектуры.

Статические страницы для ориентира:

- `autoplace-static/dealer-offer.html` - объяснение бизнес-модели;
- `autoplace-static/pricing.html` - внешняя тарифная страница.

## Связь с тестовым заданием

Выбранное тестовое задание: подписки на фильтры каталога и доставка уведомлений о совпадениях.

Минимальное ядро задания:

- `catalog_vehicles` - автомобили внутреннего каталога;
- `filter_subscriptions` - пользовательские фильтры внешнего сайта;
- `notifications` - уведомления о совпадениях;
- фоновая обработка матчей;
- идемпотентность уведомлений.

Задание про импорт автомобилей из нескольких источников используем как дополнительный ориентир для будущего демо-генератора и нормального автомобильного каталога:

- справочники марок и моделей;
- нормализация входящих данных;
- ошибки импорта не блокируют обработку остальных записей;
- повторный импорт не создает дубли.

## Принятые продуктово-архитектурные решения

На текущем этапе фиксируем следующие решения:

- главный демо-поток: система живет сама через генератор данных;
- справочники называются `makes` и `models`, как в тестовом задании;
- текущая реализация `catalog_vehicles` считается черновиком, ее можно снести и пересобрать при переходе на новую схему;
- импорт делаем в среднем варианте: raw records, mappings и Filament UI;
- REST остается для внутренних событий и технических команд;
- GraphQL/Lighthouse используется для playground, demo API и будущего UI-слоя;
- очереди сразу переводим на Redis + Horizon;
- Grafana, Prometheus и Loki добавляем позже, после стабилизации домена и генератора;
- Homepage запускается отдельным Docker-сервисом и становится точкой входа в систему;
- коммерческий слой не реализуем в БД на старте, он остается продуктовым ориентиром.

Главный демо-поток:

```text
demo generator -> source records -> import/normalization -> catalog -> matcher -> notifications -> admin/system dashboard
```

## MVP-Домены

Проект должен двигаться к модульному монолиту. На старте допустим обычный Laravel layout, но новые крупные участки проектируются как отдельные доменные зоны.

Планируемые домены:

- `Catalog` - публикация и хранение автомобилей;
- `Vehicles` - справочники марок, моделей и характеристик;
- `Subscriptions` - фильтры покупателей;
- `Notifications` - доставка и история уведомлений;
- `Imports` - демо-источники, нормализация и ошибки импорта;
- `Leads` - будущий коммерческий слой;
- `Admin` - Filament-администрирование;
- `Observability` - метрики, логи, техническая стартовая страница.

## Минимальная БД для каталога и подписок

Ближайшая целевая схема:

```text
makes

```

`makes`:

```text

```

`models`:

```text

```

`catalog_vehicles`:

```text

price
mileage
power
fuel_type

```

`filter_subscriptions`:



## Демо-генератор

Система должна уметь жить без ручного ввода данных.

Планируемый генератор:

```text
demo:tick
```

Один тик должен уметь:

- сгенерировать пачку входящих автомобилей из нескольких псевдоисточников;
- часть записей сделать валидными;
- часть записей сделать проблемными;
- импортировать нормализованные автомобили в каталог;
- запустить фоновый матчинг подписок;
- создать уведомления;
- показать результат в админке и на пользовательских экранах.

Планируемые псевдоисточники:

- `dealer_feed`;
- `auction_feed`;
- `classifieds_feed`.

Планируемая таблица для сырого потока:

```text
vehicle_source_records
- id
- source
- external_id
- raw_payload jsonb
- status pending|imported|failed
- error_code nullable
- error_message nullable
- imported_vehicle_id nullable
- processed_at nullable
- timestamps
```

## Точка входа в систему

Точкой входа после запуска Docker должен стать отдельный сервис на базе Homepage.

Homepage не встраивается в Laravel. Он запускается отдельным контейнером и показывает все доступные части MVP в одном месте.

Референс:

```text
https://github.com/gethomepage/homepage
```

Основной адрес будет выбран при настройке Docker Compose. Предпочтительный вариант:

```text
http://homepage.autoplace.127.0.0.1.sslip.io:8080
```

Если доменная маршрутизация окажется неудобной, допустим практичный вариант:

```text
http://autoplace.127.0.0.1.sslip.io:8080/system
```

Homepage должен быть не маркетинговым лендингом, а техническим центром MVP.

Примерные ссылки:

- публичный сайт AutoPlace;
- каталог;
- личный кабинет;
- Filament admin;
- Horizon;
- API Playground;
- Grafana;
- Prometheus;
- Loki;
- логи приложения;
- health-check;
- README;
- тестовый демо-сценарий.

Страница должна помогать человеку смотреть, как система живет, а не искать адреса по README.

## Capacity Target

MVP проектируется как тестовое задание с демонстрацией взрослого инженерного подхода, а не как высоконагруженный классифайд.

Базовый объем для первой версии:

```text
Пользователи:                 до 500
Автомобили в каталоге:        до 5 000
Активные подписки:            до 1 000
Новые авто в день:            100-500
Уведомления в день:           1 000-5 000
Пиковая нагрузка сайта:       5-20 RPS
```

Такой объем позволяет на старте использовать:

- PostgreSQL без отдельного поискового движка;
- JSONB-фильтр в `filter_subscriptions`;
- Laravel database queue;
- простые фоновые jobs без сложного шардирования;
- Filament как админку без отдельного back-office фронта;
- Lighthouse/GraphQL как демонстрационный API-слой.

При этом решения не должны закрывать путь к росту.

Ростовой ориентир:

```text
Пользователи:                 10 000
Автомобили в каталоге:        100 000
Активные подписки:            50 000
Новые авто в день:            5 000
Уведомления в день:           50 000
Пиковая нагрузка сайта:       100 RPS
```

Для этого уровня заранее учитываем:

- индексы по полям матчинга;
- уникальные ограничения для идемпотентности;
- возможность вынести searchable-поля из JSONB в отдельные колонки;
- батчевую обработку подписок;
- отдельные очереди для импорта, матчинга и доставки;
- наблюдаемость очередей и длительности jobs.

Формула проектирования:

```text
Строим просто под 500 пользователей,
но не принимаем решений, которые сломаются при 10 000.
```

## Load Testing

Нагрузочное тестирование нужно не для имитации highload, а чтобы проверить, что MVP не разваливается на ожидаемом демо-объеме.

Минимальные сценарии:

- создание автомобиля через API;
- массовая генерация автомобилей;
- создание подписок;
- матчинг одного нового автомобиля против набора подписок;
- повторная обработка автомобиля без дублей уведомлений;
- чтение каталога и уведомлений через UI/API.

Инструменты:

- `k6` для HTTP smoke/load-тестов;
- Artisan-команды для генерации объемных seed-данных;
- Prometheus/Grafana для наблюдения метрик;
- Loki для просмотра логов ошибок.

Целевые значения для MVP:

```text
Создание автомобиля через API:       p95 < 300ms
Постановка matcher job в очередь:    p95 < 500ms
Матчинг одного авто на 1k подписок:  p95 < 2s
Появление уведомления:               до 60 секунд
Повторная обработка:                 0 дублей уведомлений
```

## Политика API и очередей

Для MVP используем простое правило:

```text
GET-запросы отвечают напрямую.
POST/PUT/DELETE-запросы не выполняют тяжелую бизнес-логику в HTTP-запросе.
Долгие операции выполняются через очередь.
```

GET-запросы:

- читают данные напрямую из PostgreSQL;
- всегда используют пагинацию для списков;
- защищаются rate limit от ботов и частых запросов;
- не запускают тяжелые вычисления;
- при необходимости позже получают кэширование.

POST/PUT/DELETE-запросы:

- валидируют входные данные сразу;
- сохраняют минимальный факт или команду в БД;
- ставят job в очередь;
- быстро возвращают ответ клиенту.

Например, создание автомобиля через API должно работать так:

```text
1. принять и провалидировать данные;
2. создать запись автомобиля;
3. поставить задачу матчинга в очередь;
4. вернуть 202 Accepted.
```

Матчинг подписок, доставка уведомлений, импорт, нормализация и демо-генерация выполняются фоновыми jobs.

Целевые ориентиры:

```text
GET-запросы:                         p95 < 300ms
POST с постановкой в очередь:        p95 < 500ms
Матчинг одного авто на 1k подписок:  p95 < 2s в фоне
Появление уведомления у пользователя: до 60 секунд
```

Это не жесткое обещание времени до реализации. Это performance budget: после появления метрик и нагрузочных тестов значения проверяются и корректируются.

## Очереди и фоновые процессы

Матчинг не должен выполняться в HTTP-запросе.

Фоновые задачи:

- обработка события нового автомобиля;
- матчинг подписок;
- доставка уведомлений;
- демо-генерация;
- импорт и нормализация source records.

На старте используем Redis + Horizon, чтобы очередь была видимой частью демо и ее можно было наблюдать через техническую точку входа.

## Frontend

Основной стек:

- Inertia.js;
- React;
- TypeScript;
- Tailwind CSS 4;
- shadcn/ui-подход к компонентам.

Принципы:

- страницы собираются из переиспользуемых секций;
- UI-примитивы выносятся в `components/ui`;
- бизнес-компоненты выносятся в feature-зоны;
- статические прототипы могут жить в `autoplace-static`, но продуктовые экраны должны постепенно переноситься в Inertia.

## Docker и локальный запуск

Проект должен запускаться из чистого окружения через Docker Compose.

Текущие адреса:

```text
http://autoplace.127.0.0.1.sslip.io:8080
http://admin.autoplace.127.0.0.1.sslip.io:8080
```

Портовые решения должны быть практичными:

- PostgreSQL наружу на `5433`, чтобы не конфликтовать с локальным PostgreSQL;
- Caddy наружу на `8080`, чтобы не конфликтовать с занятым портом `80`;
- Vite наружу на `5173`.

## Инженерные принципы

- Сначала доменная ясность, потом UI.
- Не смешивать импорт, каталог, подписки, уведомления и будущие лиды в одну модель.
- Идемпотентность должна быть на уровне БД, а не только в коде.
- Демо-данные должны быть воспроизводимыми.
- Ошибки должны классифицироваться.
- README должен объяснять запуск, но стартовая страница должна объяснять систему.
- Временные решения допустимы, если явно понятно, как их заменить.

## Этапы реализации

### Этап 0. Фиксация направления

Цель: закрепить MVP-вектор и не продолжать разработку поверх черновой модели.

Работы:

- зафиксировать `docs/mvp.md`;
- оставить коммерческий слой только как продуктовый ориентир;
- считать текущую схему `catalog_vehicles` черновиком;
- договориться, что основная точка входа будет отдельным Homepage-сервисом;
- договориться, что очереди переводятся на Redis + Horizon.

Результат:

- есть документированный план развития;
- понятно, какие решения временные;
- понятно, что реализуем дальше и что не трогаем.

### Этап 1. Инфраструктурный каркас

Цель: сделать проект удобным для демонстрации и наблюдения.

Работы:

- добавить Redis;
- подключить Laravel Horizon;
- перевести очереди на Redis;
- добавить Homepage отдельным Docker-сервисом;
- настроить Caddy-маршруты для Homepage, сайта, админки и Horizon;
- обновить README с новой точкой входа.

Результат:

- после `docker compose up --build` есть единая стартовая панель;
- видны сайт, админка, Horizon и технические ссылки;
- фоновые процессы можно наблюдать.

### Этап 2. Новая доменная БД каталога

Цель: перейти от черновой схемы к минимальной, но нормальной автомобильной модели.

Работы:

- пересобрать миграции под `makes`, `models`, `catalog_vehicles`, `filter_subscriptions`, `notifications`;
- добавить связи `makes -> models -> catalog_vehicles`;
- сохранить контракт фильтров с ключами `make_id` и `model_id`;
- добавить seed для марок, моделей, автомобилей, подписок и уведомлений;
- сохранить уникальный индекс `notifications(subscription_id, vehicle_id, type)`.

Результат:

- БД соответствует тестовому заданию и MVP-вектору;
- справочники марок и моделей становятся частью системы;
- идемпотентность уведомлений закреплена на уровне БД.

### Этап 3. Доменная логика и фоновые процессы

Цель: привести matcher, события и jobs к новой схеме.

Работы:

- обновить `VehicleFilterMatcher`;
- обновить job обработки нового автомобиля;
- отделить матчинг от доставки уведомлений;
- перевести обработку на Redis queue;
- добавить понятные статусы и логи для фоновых jobs;
- обновить feature/integration tests.

Результат:

- создание автомобиля быстро ставит работу в очередь;
- matcher работает по новой БД;
- повторная обработка не создает дубли;
- тесты покрывают ключевой workflow.

### Этап 4. Filament-админка

Цель: сделать админку инструментом наблюдения за системой, а не только CRUD.

Работы:

- пересобрать ресурсы Filament под `makes`, `models`, `catalog_vehicles`, `filter_subscriptions`, `notifications`;
- добавить страницы для raw source records и mappings после появления импорта;
- обновить dashboard-виджеты;
- добавить быстрые действия: создать демо-авто, запустить матчинг, поставить подписку на паузу;
- оставить старые черновые экраны только если они не мешают.

Результат:

- админ видит состояние каталога, подписок, уведомлений и очередей;
- демо-сценарий можно пройти без ручного SQL;
- Filament отражает реальную доменную модель.

### Этап 5. Демо-генератор и импорт

Цель: сделать систему живой без ручного ввода.

Работы:

- добавить таблицу `vehicle_source_records`;
- добавить таблицу mappings для терминов источников;
- реализовать псевдоисточники `dealer_feed`, `auction_feed`, `classifieds_feed`;
- добавить команду `demo:tick`;
- генерировать валидные и ошибочные записи;
- импортировать нормализованные автомобили;
- запускать matcher после успешного импорта.

Результат:

- система сама получает новые автомобили;
