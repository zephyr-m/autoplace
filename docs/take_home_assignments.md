# Take-Home Assignment (на выбор 1 из 2 заданий)

**Срок выполнения:** 2–3 дня

**Технологический стек:** Laravel · React + TypeScript + Inertia.js · Docker (`docker compose`) · PostgreSQL

**Цель:** реализовать функциональность, демонстрирующую системное проектирование доменной модели, корректную обработку ошибок и качество инженерных решений

---

## Общие требования

1. Приложение запускается из чистого окружения командой `docker compose up --build` (или аналогичной — необходимо описать в  README.md)
2. Исходные данные для демо воспроизводимы через seed/fixtures в репозитории
3. Обязательно реализовать: валидацию входных данных на стороне сервера, корректную обработку исключений
4. Ключевые бизнес-сценарии покрыты автоматическими тестами — минимум один test suite уровня feature/integration
5. UI включает все экраны, необходимые для полноценной демонстрации (никаких заглушек)
6. README.md содержит инструкции по сборке, запуску и прогону тестов

## Задание 1. Импорт автомобилей из нескольких источников с нормализацией марки/модели

### Контекст

Компания получает данные об автомобилях из нескольких внешних источников. У каждого источника — свой формат и своя номенклатура: названия марок и моделей в источниках не совпадают с каноническими названиями в доменном каталоге (например, из-за локализации, аббревиатур или опечаток)

Необходимо реализовать механизм импорта, который приводит записи из источников к единой доменной модели

### Входные данные

Реализуйте интеграцию с **тремя источниками** с разными форматами данных. Формат, протокол и способ подключения — на ваше усмотрение. Каждый источник должен содержать:

- идентификатор автомобиля в рамках источника,
- названия марки и модели в терминах источника,
- не менее 3–5 характеристик автомобиля

Данные для демо-запуска должны быть воспроизводимы из репозитория

### Ожидаемое поведение импорта

1. Запись, которую не удалось нормализовать, **не блокирует** обработку остальных — ошибка фиксируется, импорт продолжается
2. Повторный запуск не создаёт дубликаты и корректно обновляет изменяемые поля
3. Классифицируйте ошибки (не всё, что пошло не так — одно и то же) и добавьте соотвествующее поведение: логгирование/уведомление/Sentry

### Доменная модель (минимум)

```
makes          — id, name
models         — id, make_id, name
vehicles       — id, make_id, model_id, <поля источника по вашему усмотрению>, attributes (JSON)
```

Структуру для хранения маппингов между терминами источников и доменными сущностями спроектируйте самостоятельно

### UI (минимум)

- Экран просмотра автомобилей с фильтрацией по марке и модели
- Экран для управления маппингами: просмотр, добавление, редактирование

### Тестирование

Покройте тестами минимум:

- повторный запуск не создаёт дубликаты,
- запись без маппинга не роняет импорт,
- нормализация даёт корректный результат на фиксированных примерах

## Задание 2. Подписки на фильтры каталога и доставка уведомлений о совпадениях

### Контекст

Имеется **две независимые системы**:

- **Внутренняя система** (back-office) — управляет каталогом автомобилей, предоставляет API
- **Внешняя система** (сайт) — пользовательский интерфейс. Получает данные из внутренней системы через API

Пользователи сайта могут подписаться на поисковый фильтр. Когда во внутренней системе появляется автомобиль, удовлетворяющий условиям подписки, пользователь должен получить уведомление

Обе системы реализуются в рамках одного git репозитория и запускаются через `docker compose`

### Параметры фильтра (минимум 10)

Фильтр должен поддерживать следующие параметры в произвольных комбинациях. Все параметры должны быть валидированы и типизированы:

| Параметр | Тип |
|---|---|
| `make_id` | integer |
| `model_id` | integer |
| `min_price` / `max_price` | Integer |
| `min_mileage` / `max_mileage` | integer |
| `min_power` / `max_power` | integer |
| `fuel_type` | enum |
| `year_from` / `year_to` | integer |

### Ожидаемое поведение

1. Пользователь создаёт подписку на фильтр на внешнем сайте
2. Во внутренней системе появляется новый автомобиль
3. Внешняя система узнаёт об этом событии и проверяет активные подписки на совпадение
4. При совпадении пользователь получает уведомление. **Повторная обработка того же автомобиля не создаёт дубль уведомления**

Как именно внешняя система узнаёт о событиях из внутренней — архитектурное решение на ваше усмотрение

### Доменная модель (минимум)

**Внутренняя система:**

```
catalog_vehicles  — поля, достаточные для матчинга по всем параметрам фильтра
```

**Внешняя система:**

```
filter_subscriptions  — user_identifier, filter (JSON), status (active|paused)
notifications         — subscription_id, vehicle_id, type, payload (JSON)
```

Дополнительные сущности, необходимые для корректной работы, — на ваше усмотрение

### Эмуляция события в тестовом окружении

Предоставьте способ инициировать событие «новый автомобиль в каталоге» — через endpoint, команду Artisan или seed. Опишите его в README.md

### UI (минимум)

**Внешняя система:**

- Форма создания подписки (страница фильтрации со всеми параметрами фильтра + кнопка подписки на фильтр)
- Экран «Мои подписки» с возможностью удаления подписки
- Экран / UI блок со списком всех полученных уведомлений

### Архитектурные требования

- Логика матчинга фильтра должна быть отделена от механизма доставки уведомлений
- Фоновая обработка совпадений обязательна: матчинг не выполняется в рамках HTTP-запроса
- Idempotency матчей обязательна

### Тестирование

Покройте тестами минимум:

- корректность матчинга на фиксированных наборах автомобилей (включая 2–3 комбинации параметров),
- повторная обработка не создаёт дублей уведомлений
